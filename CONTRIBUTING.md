# Contributing to OB Knowledge Graph

Thank you for your interest in contributing to this project! We welcome contributions from the community.

## Code of Conduct

Please be respectful and constructive in all interactions. We are committed to providing a welcoming and inclusive environment for all contributors.

## How to Contribute

### Reporting Issues

If you find a bug or have a suggestion, please open an issue on GitHub:

1. Check if the issue already exists
2. Provide a clear description of the problem
3. Include steps to reproduce (for bugs)
4. Mention your browser and operating system

### Improving Translations

We welcome improvements to the Chinese and Japanese translations:

1. Fork the repository
2. Edit the relevant language strings in `index.html`
3. Test the changes in your browser
4. Submit a pull request with a clear description of your changes

### Suggesting Features

Have an idea for a new feature? Please:

1. Open an issue describing your idea
2. Explain the use case and benefits
3. Discuss with the community before implementing

### Submitting Code Changes

1. Fork the repository
2. Create a new branch for your feature: `git checkout -b feature/your-feature-name`
3. Make your changes
4. Test thoroughly in multiple browsers
5. Commit with clear, descriptive messages
6. Push to your fork
7. Submit a pull request

### Pull Request Guidelines

- Keep commits atomic and focused
- Write clear commit messages
- Update documentation as needed
- Test in Chrome, Firefox, Safari, and Edge
- Ensure code follows the existing style
- Add comments for complex logic
- Update the README if adding new features

## Development Setup

1. Clone the repository:
   ```bash
   git clone https://github.com/yourusername/OB-Knowledge-Graph.git
   cd OB-Knowledge-Graph
   ```

2. Start a local server:
   ```bash
   python3 -m http.server 8000
   ```

3. Open http://localhost:8000 in your browser

4. Make your changes and test

## Code Style

- Use meaningful variable names
- Add comments for non-obvious code
- Keep functions focused and modular
- Use consistent indentation (2 spaces for HTML/CSS, 4 for JavaScript)

## Documentation

- Update README.md for user-facing changes
- Update ATTRIBUTION.md if modifying source attribution
- Add comments to complex code sections
- Keep documentation accurate and up-to-date

## Testing

- Test in multiple browsers
- Test on different screen sizes
- Verify all interactive features work correctly
- Check that language switching works properly
- Ensure search functionality is responsive

## Questions?

Feel free to open an issue or discussion if you have questions about contributing.
